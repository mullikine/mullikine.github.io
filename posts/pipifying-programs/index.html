The goal

To be able to imbue any interactive terminal program with the ability to take stdin, return stdout or any combination of such and to make this available to the user without sacrificing the ability to interact.

The above articles are related because I also want to be able to automate such programs to any degree and build emacs modes for them, without sacrificing the pipability of such programs.

The motivation

It’s useful to be able to treat applications in the terminal as being pipable (accepting stdin and returning stdout). An example of a useful utility which takes stdin, returns stdout while providing interactivity via a user interface is the vipe command.

vipe enables you edit data with your editor as it moves through a bash pipe. But how does it work?

How vipe and vim work

By design vipe delegates editing to vim. This is because vim is designed to accept input from a pipe and then retake control of the input file descriptor by connecting it to the tty.

vim is unusual among terminal utilities in that it has the ability to accept stdin and then display a user interface for editing. This is unusual because there is only one input file descriptor (0) and among programs it is usually used only for one purpose: either piping into the program or reading the user’s keyboard input from the tty.

For example,

1

echo hi | vim -

However, vim - is not designed to pipe back out of itself. The output is always used to be the tty.

vipe completes the pipability of vim by allowing you to pipe back out of it. It does this by telling vim to edit a temporary file, reassigning both fd 1 and 2 to /dev/tty.

It’s not exactly that simple though. Not only must file descriptors 1, 2 and 3 point to /dev/tty or the proper tty/pts that /dev/tty itself points to, but they also must point to that tty via the same pointer.

You can ensure this by redirecting one to the other rather than directly to the tty. For example,

1
2
3

exec 1>/dev/tty
exec 2>&1
exec 0<&1

Pipifying interactive terminal commands then is simply a matter of juggling these file descriptors and using temporary files for soaking the data as it passes through.

Useful bash script and functions we need to proceed

is_tty and stdin_exists

is_tty will tell you if the command (in this case, vipe) was spawned from a tty.

stdin_exists will tell you if the command has input piped into it.

Under the hood, what both of these tests are actually determining is if the file descriptor currently points to the tty, but it’s another way of asking if it has been reassigned (probably to a pipe).

The tm-tty script

Here is the heart of the mountain.

This script determines in a smart way what the tty associated with your current program is.

It’s not always easy to determine; for example, if you are running dwarf fortress inside tmux, inside an expect script inside of emacs, inside of another expect script, inside of tmux. In that case you probably want the innermost tmux’s tty – that is, unless you have specified the TTY environment variable and so wish to override that.

This script first checks for that environment variable. If that doesn’t exist then it will run the tty command. If that fails it will test /dev/tty to see if a file descriptor can be assigned to it without any problems. If that fails it will ask the innermost tmux for the pty.

#!/bin/bash

{
    test -n "$TTY" && echo "$TTY"
} || {
    ttymaybe="$(tty 2>&1)"
    if [[ "${ttymaybe:0:1}" == "/" ]]; then
        printf -- "%s\n" "$ttymaybe"
        :
    else
        false
    fi
} || {
    {
       exec 5>/dev/tty && echo /dev/tty
    } 2>/dev/null
} || {
    TTY="$(tmux display-message -p '#{pane_tty}')"
    echo "$TTY"
}

You then use the script as follows.

1

exec <`tm-tty`

That is much more reliable than doing exec /dev/tty, which often fails.

#!/bin/bash
export TTY


is_tty() {
    [[ -t 1 ]]
}

stdin_exists() {
    ! [ -t 0 ]
}

if is_tty; then
    /usr/bin/vipe "$@"
else
    if stdin_exists; then
        /usr/bin/vipe "$@"
    else
        tf_zle="$(mktemp ${TMPDIR}/tf_zleXXXXXX || echo /dev/null)"
        exec 3>&1
        exec 1>`tm-tty`
        vim "$tf_zle"
        exec 1>&3
        cat "$tf_zle"
    fi
fi

stdin_exists() {
    ! [ -t 0 ]
}

# Collect the stdin
if stdin_exists; then
    tempfile="$(0</dev/null mktemp emacs-stdin-$USER.XXXXXXX --tmpdir)"
    cat > "$tempfile"
fi

# Find the current tty and assign input to it, so we can collect input from the user
: ${TTY:="$(tm-tty)"}
exec <"$TTY"

# save the stdout and stderr file descriptors to 3 and 4
exec 3>&1
exec 4>&2

# assign stdout and stderr to the same tty as the input tty
exec 1>&0
exec 2>&0

# We then run our command which has full tty access. It could be emacs for example
emacs -nw "$tempfile"

# Restore stdout and stderr to their regular output file descriptors
# Whatever they used to point to
exec 1>&3
exec 2>&4

# If there was stdin then cat it out to the terminal
# The pipeline around emacs is then completed
if test -f "$tempfile"; then
    cat "$tempfile"
fi
